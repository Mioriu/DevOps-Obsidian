# Саммари по мониторингу Linux

## Зачем нужен мониторинг

**Без мониторинга вы узнаёте о проблемах, когда уже поздно.** Три истории:

- Процесс с утечкой памяти съедал по 100MB/час → через 3 дня сервер упал
    
- Логи nginx заняли 100% диска в пятницу вечером → потерянные выходные продажи
    
- SQL-запрос грузил CPU на 100% только при определённых условиях → недели поиска бага
    

**Четыре кита мониторинга:** CPU, память, диски, сеть.

---

## top — первый помощник

bash

top

### Ключевые строки

**Load average:** 0.45, 0.67, 0.72 — средняя загрузка за 1, 5, 15 минут.

- **Правило:** load average не должен надолго превышать количество ядер CPU (`nproc`)
    

**CPU:** `us` (пользовательские), `sy` (системные), `id` (простой), `wa` (ожидание диска)

- `wa` > 10-20% — диски не справляются!
    

**Память:** смотрите на `available`, а не на `free`! Linux использует свободную память для кэша.

**Swap used > 0** — системе не хватает памяти, производительность падает.

### Горячие клавиши top

- `M` — сортировка по памяти
    
- `P` — сортировка по CPU
    
- `1` — показать каждое ядро
    
- `c` — полная командная строка
    
- `k` — убить процесс

## htop — top на стероидах

bash

sudo apt install htop   # установка
htop

**Преимущества:**

- Цветные полоски загрузки CPU для каждого ядра
    
- Наглядная шкала памяти
    
- Дерево процессов (F5)
    
- Фильтр (F4) и поиск (F3)
    
- Убийство процессa (F9) с выбором сигнала

## Память — не дайте ей закончиться

bash

free -h

**Пример вывода:**

text

               total  used  free  buff/cache  available
Mem:            15Gi  8.0Gi 2.1Gi   5.9Gi       7.1Gi
Swap:          8.0Gi     0B 8.0Gi

**Главное правило:** смотрите на `available`, а не на `free`!

**Когда бить тревогу:**

- available < 10% от total
    
- swap used > 0 при нормальной работе
    
- swap активно растёт
    

**Мониторинг в реальном времени:**

bash

watch -n 1 free -h     # обновление каждую секунду
watch -d free -h       # подсветка изменений

**Детальная информация:**

bash

cat /proc/meminfo
vmstat -s              # более читаемый формат

## Диски — место и скорость

### Сколько места осталось — df

bash

df -h                  # человекочитаемый формат
df -i                  # использование inode (для мелких файлов)

**Use% > 80%** — пора чистить, **> 90%** — срочно чистить!

### Кто съел место — du

bash

du -h /var | sort -hr | head -20    # топ-20 папок в /var
find / -type f -size +100M -exec ls -lh {} \; 2>/dev/null   # файлы >100MB

**Интерактивный анализ:** `sudo apt install ncdu && sudo ncdu /`

### Скорость дисков — iostat

bash

iostat -x 1            # расширенная статистика, обновление каждую секунду

**Ключевые метрики:**

- `%util` — загрузка диска (>70% нагружен, >90% перегружен)
    
- `await` — время ожидания (>20ms — медленно)
    
- `%iowait` — CPU ждёт диск (>20% — проблема)
    

### Кто грузит диски — iotop

bash

sudo apt install iotop
sudo iotop             # показывает процессы, активно использующие диски

## Сеть — что и куда течёт

### Открытые порты — ss

bash

ss -tuln               # слушающие порты (TCP/UDP)
ss -tn                 # активные соединения

### Трафик в реальном времени

bash

sudo apt install iftop
sudo iftop -i eth0     # трафик по соединениям
sudo apt install nload
nload                  # графики загрузки интерфейса
sudo apt install nethogs
sudo nethogs           # трафик по процессам

### Статистика интерфейсов

bash

ip -s link show        # счётчики ошибок, дропов

**Важно:** `errors` и `dropped` должны быть 0.

## vmstat — швейцарский нож

bash

vmstat 2               # обновление каждые 2 секунды

**Вывод и расшифровка:**

text

procs ---------memory-------- ---swap-- -----io---- -system-- ------cpu-----
 r  b   swpd   free  buff cache  si  so   bi   bo   in   cs us sy id wa st
 1  0      0 2195584 165432 6.1G  0   0    5   23   145  298  5  2 93  0  0

- `r` — очередь на CPU (если > ядер — перегрузка)
    
- `b` — процессы, ждущие I/O (если часто >0 — проблемы с дисками)
    
- `si/so` — swap in/out (если не 0 — не хватает памяти)
    
- `wa` — ожидание I/O (>20% — диски тормозят)
    
- `cs` — переключения контекста (>100000 — слишком много)

## Пошаговая диагностика (шпаргалка)

1. **Запускаем top** — смотрим load average. Если больше ядер или близко к ним — копаем дальше.
    
2. **Проверяем память** `free -h` — если available < 10% или swap used > 0 — проблемы с памятью.
    
3. **Проверяем диски** `iostat -x 1` — если %util > 90% или await > 50ms — диски не справляются.
    
4. **Смотрим сеть** `iftop` — может, кто-то качает терабайты?
