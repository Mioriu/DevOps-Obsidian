# Саммари по логированию в Linux

## Где живут логи и зачем они нужны

**Основная директория:** `/var/log/` — стандартное место хранения всех логов.

**Три главные задачи логов:**

1. **Расследование проблем** — понять, что сломалось
    
2. **Безопасность** — кто и когда пытался войти
    
3. **Аудит и соответствие** — требования законодательства
    

**Ключевые системные логи:**

- `/var/log/syslog` (Ubuntu) или `/var/log/messages` (RHEL) — почти все системные сообщения
    
- `/var/log/auth.log` (Ubuntu) или `/var/log/secure` (RHEL) — попытки входа, sudo
    
- `/var/log/kern.log` — сообщения ядра (драйверы, оборудование)
    

**Логи сервисов:** `/var/log/nginx/access.log` (кто и что запрашивал), `/var/log/nginx/error.log` (ошибки веб-сервера)

## Две системы логирования

### 1. Классическая: rsyslog

Пишет логи в **текстовые файлы**. Работает по принципу почтового отделения:

- **Facility (источник):** кто отправил (auth, cron, kern, mail, local0-local7)
    
- **Priority (важность):** отладка (debug) → информация (info) → предупреждение (warning) → ошибка (err) → критическая (crit) → авария (emerg)
    
- Правила: `facility.priority действие` (например, `auth.* /var/log/auth.log`)
    

### 2. Современная: systemd-journald

Хранит логи в **структурированной базе данных** с метаданными. Преимущества:

- Поиск по любому полю
    
- Автоматическая ротация
    
- Централизованное хранение
    

**Как работают вместе:** journald собирает всё → может передавать в rsyslog → rsyslog пишет в текстовые файлы.

## Работа с логами через journalctl

**Навигация:** пробел (вниз), b (вверх), g (в начало), G (в конец), /текст (поиск), q (выход)

**Основные команды:**

- `journalctl -e` — открыть с конца (свежие события)
    
- `journalctl -f` — следить в реальном времени
    
- `journalctl -n 50` — последние 50 строк
    

**Фильтрация по времени:**

- `journalctl --since today` — за сегодня
    
- `journalctl --since "1 hour ago"` — за последний час
    
- `journalctl --since "2025-07-16 09:00" --until "2025-07-16 10:00"` — конкретный период
    

**Фильтрация по важности:**

- `journalctl -p err` — только ошибки и выше
    
- Уровни: emerg (0), alert (1), crit (2), err (3), warning (4), notice (5), info (6), debug (7)
    

**Фильтрация по сервису:**

- `journalctl -u nginx.service` — логи конкретного сервиса
    
- `journalctl -u nginx -u mysql` — несколько сервисов
    

**Поиск:**

- `journalctl -g "error"` — простой поиск по тексту
    
- `journalctl -g "192\.168\.[0-9]+\.[0-9]+"` — с регулярным выражением
    

**Форматы вывода:**

- `-o cat` — только сообщения
    
- `-o verbose` — все поля
    
- `-o json` — для скриптов
    

## Настройка rsyslog

**Где правила:**

- Главный файл: `/etc/rsyslog.conf`
    
- Дополнительные: `/etc/rsyslog.d/*.conf`
    

**Создание своего правила:**

text

# Все сообщения от myapp в отдельный файл
:programname, isequal, "myapp" /var/log/myapp/app.log
# Критические ошибки в отдельный файл
if $programname == 'myapp' and $syslogseverity <= 3 then {
    action(type="omfile" file="/var/log/myapp/critical.log")
}
# Прекратить обработку
:programname, isequal, "myapp" stop

**Полезные возможности:**

- Отправка на удалённый сервер: `*.* @192.168.1.100:514` (UDP) или `*.* @@192.168.1.100:514` (TCP)
    
- Фильтрация по содержимому: `:msg, contains, "ERROR" /var/log/errors.log`
    
- Динамические имена: `$template DailyLog,"/var/log/myapp/%$YEAR%-%$MONTH%-%$DAY%.log"`
    

## Управление размером логов в journald

**Проверка размера:** `journalctl --disk-usage`

**Настройки в `/etc/systemd/journald.conf`:**

text

SystemMaxUse=1G          # Максимум места на диске
SystemKeepFree=500M      # Минимум свободного места
MaxRetentionSec=1month   # Удалять логи старше месяца
SystemMaxFileSize=100M   # Максимальный размер одного файла
Compress=yes             # Сжимать старые файлы

**Ручная очистка:**

- `journalctl --vacuum-time=2weeks` — оставить последние 2 недели
    
- `journalctl --vacuum-size=500M` — оставить только 500MB
    
- `journalctl --vacuum-files=3` — оставить 3 последних файла
    

## Ротация логов через logrotate

**Зачем:** чтобы логи не заполнили весь диск. Автоматически переименовывает, создаёт новые, сжимает старые, удаляет совсем старые.

**Где настройки:**

- Главный файл: `/etc/logrotate.conf`
    
- Индивидуальные: `/etc/logrotate.d/*`
    

**Ключевые директивы:**

- `daily/weekly/monthly` — частота ротации
    
- `rotate 4` — хранить 4 старых версии
    
- `compress` — сжимать gzip
    
- `delaycompress` — сжимать при следующей ротации
    
- `size 100M` — ротировать при достижении размера
    
- `missingok` — не ругаться, если файла нет
    
- `notifempty` — не ротировать пустые
    
- `create 644 app app` — создать новый файл с правами
    
- `postrotate ... endscript` — скрипт после ротации
    

**Пример для nginx:**

text

/var/log/nginx/*.log {
    daily
    missingok
    rotate 14
    compress
    delaycompress
    notifempty
    create 640 nginx adm
    sharedscripts
    postrotate
        [ -f /var/run/nginx.pid ] && kill -USR1 `cat /var/run/nginx.pid`
    endscript
}

**Команды logrotate:**

- `logrotate -d /etc/logrotate.d/myapp` — проверка (без выполнения)
    
- `logrotate -fv /etc/logrotate.d/myapp` — принудительная ротация с подробным выводом
    

**Где хранится состояние:** `/var/lib/logrotate/status` — дата последней ротации каждого файла

## Практические приёмы

**Быстрая проверка проблем с доступом:**

text

tail -20 /var/log/auth.log
journalctl -u ssh.service -p err --since "24 hours ago"

**Мониторинг в реальном времени:**

text

tail -f /var/log/nginx/access.log
journalctl -f -u nginx

**Расследование инцидента:**

text

journalctl --since "yesterday 14:00" --until "yesterday 15:00" -p crit

**Настройка под разные сценарии:**

- Обычный ПК: `SystemMaxUse=500M`, `MaxRetentionSec=1week`
    
- Сервер разработки: `SystemMaxUse=2G`, `MaxRetentionSec=1month`
    
- Production: `SystemMaxUse=10G`, `MaxRetentionSec=6months`, `Compress=yes`
