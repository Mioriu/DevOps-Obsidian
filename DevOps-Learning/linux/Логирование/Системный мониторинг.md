# Часть 1: Зачем нужен мониторинг и что может пойти не так

Представьте, что вы водитель автомобиля. У вас есть приборная панель: спидометр, тахометр, указатель топлива, температура двигателя. Без них вы не знаете, всё ли в порядке с машиной. То же самое с сервером — без мониторинга вы слепы.

## Что может быть без мониторинга

- **История 1 - медленная смерть**  
    Клиентский сервис работал всё медленнее. Владельцы думали: "_Просто много клиентов_". Через 3 дня сервис упал. Причина? Процесс с утечкой памяти съедал по 100MB в час. **За 72 часа — вся память кончилась.**
- **История 2 - пятничный сюрприз**  
    Пятница, 17:00. Все уходят домой. В 18:00 сайт перестаёт работать. Причина? Логи nginx заняли 100% диска. **Потери**: выходные продажи и нервы администратора.
- **История 3 - загадочные тормоза**  
    База данных периодически "_подвисала_" на 5-10 секунд. Разработчики искали баг. **Оказалось**: один SQL-запрос загружал CPU на 100%, но **только когда клиент из Австралии заходил в определённый раздел.**

**Вывод:** Без мониторинга вы узнаёте о проблемах, когда уже поздно!

### Что нужно мониторить?

Четыре кита мониторинга:

1. **CPU** — не перегревается ли "мозг" сервера?
2. **Память** — хватает ли "оперативки"?
3. **Диски** — есть ли место и не тормозят ли?
4. **Сеть** — всё ли доходит и с какой скоростью?

- `load average: 0.45, 0.67, 0.72` — средняя загрузка (самое важное, как известно из уроков ранее)

## Load Average — ключевая метрика

Повторим: Load Average — это среднее количество процессов, которые либо выполняются, либо ждут выполнения.

### **У сервера обычно несколько ядер CPU**

**Узнаём количество ядер:**

```bash
nproc
```

**Пример вывода:**

```undefined
4
```

Значит, у вас 4 ядра. Теперь **интерпретируем load average:**

- Load 4.00 = 100% загрузка системы
- Load 2.00 = 50% загрузка системы
- Load 8.00 = Лютый перегруз

Load average не должен надолго превышать количество ядер. Но и когда он близок к ним - надо задуматься: проверяем ядра, диск, память, сеть.

## Строка Tasks — что делают процессы

```lua
Tasks: 245 total,   2 running, 243 sleeping,   0 stopped,   0 zombie
```

- `245 total` — всего процессов в системе
- `2 running` — выполняются прямо сейчас
- `243 sleeping` — спят (ждут чего-то)
- `0 stopped` — приостановлены (Ctrl+Z)
- `0 zombie` — зомби (должно быть 0!)

**Что такое процесс-зомби?** Это процесс, который завершился, но родитель не забрал его код возврата. Несколько зомби — не страшно. Сотни — проблема в приложении, которое их плодит.

## Строка CPU — куда уходит процессорное время

```scss
%Cpu(s):  5.3 us,  2.1 sy,  0.0 ni, 91.8 id,  0.5 wa,  0.0 hi,  0.3 si,  0.0 st
```

Расшифровка каждого значения:

- `us` (user) = 5.3% — ваши программы (веб-сервер, база данных)
- `sy` (system) = 2.1% — ядро системы (файлы, сеть, драйверы)
- `ni` (nice) = 0.0% — процессы с изменённым приоритетом
- `id` (idle) = 91.8% — простой (CPU отдыхает!)
- `wa` (wait) = 0.5% — ожидание диска (важный показатель!)
- `hi` (hardware interrupts) = 0.0% — аппаратные прерывания
- `si` (software interrupts) = 0.3% — программные прерывания
- `st` (stolen) = 0.0% — украдено гипервизором (для виртуалок)

**На что обращать внимание:**

- `wa` больше 10-20% — диски не справляются!
- `st` больше 5% — проблемы с виртуализацией
- `us + sy` близко к 100% — CPU перегружен

## Строки памяти — хватает ли RAM

```yaml
MiB Mem :  16384.0 total,   2145.3 free,   8234.7 used,   6004.0 buff/cache
MiB Swap:   8192.0 total,   8192.0 free,      0.0 used.   7234.5 avail Mem
```

### **Память (Mem):**

- `total` = 16GB — всего оперативки
- `free` = 2.1GB — совсем не используется
- `used` = 8.2GB — используется программами
- `buff/cache` = 6GB — кэш (можно освободить!)
- `available` = 7.2GB — реально доступно

**Важно:** Linux использует свободную память для кэша. Смотрите на available, а не на free!

### **Swap (подкачка):**

- `total` = 8GB — размер swap-раздела
- `used` = 0 — отлично! Swap не используется

Если **swap used > 0** — системе не хватает памяти, производительность падает!

## Список процессов — кто что делает

```iecst
  PID USER      PR  NI    VIRT    RES    SHR S  %CPU  %MEM     TIME+ COMMAND
 2345 www-data  20   0  456780 123456  67890 S  12.3   0.7   1:23.45 nginx
```

Расшифровка колонок:

- `PID` — уникальный номер процесса
- `USER` — от чьего имени работает
- `PR` — приоритет (20 = обычный)
- `NI` — nice value (-20 до +19)
- `VIRT` — виртуальная память (может быть огромной)
- `RES` — реальная память (это важно!)
- `SHR` — разделяемая память
- `S` — состояние (S=спит, R=работает, Z=зомби)
- `%CPU` — процент использования CPU
- `%MEM` — процент использования памяти
- `TIME+` — общее время работы
- `COMMAND` — имя программы

## Горячие клавиши top — управление

top — интерактивная программа. Вот самые полезные клавиши:

**Сортировка:**

- `M` — по памяти (Memory) — найти жадные процессы
- `P` — по CPU (Processor) — кто грузит процессор
- `T` — по времени работы (Time)
- `N` — по PID (Number)

**Отображение:**

- `1` — показать загрузку каждого ядра отдельно
- `t` — переключить вид CPU
- `m` — переключить вид памяти
- `c` — показать полную командную строку
- `V` — древовидный вид процессов

**Действия:**

- `k` — убить процесс (введите PID)
- `r` — изменить приоритет (renice)
- `u` — показать процессы конкретного пользователя
- `h` — справка
- `q` — выход

## Практический пример анализа

**Вы запустили top и видите:**

```shell
load average: 8.45, 7.67, 6.72
%Cpu(s): 45.3 us, 12.1 sy, 0.0 ni, 2.8 id, 39.5 wa
```

**Анализ:**

1. Load 8 при 4 ядрах = двойная перегрузка!
2. CPU: 45% user + 12% system = 57% загрузка
3. Но! 39.5% wa (wait) = **процессор ждёт диски!**
4. **Вывод: проблема не в CPU, а в медленных дисках**

# Часть 3: htop — top на стероидах

htop — это как переход с чёрно-белого телевизора на цветной. Те же данные, но намного удобнее!

## Установка htop

```bash
sudo apt install htop    # Debian/Ubuntu
sudo yum install htop    # RHEL/CentOS
```

## Запуск и первое впечатление

```nginx
htop
```

### **Что вы увидите:**

- Цветные полоски загрузки CPU для каждого ядра
- Наглядная шкала памяти
- Древовидная структура процессов
- Подсказки внизу экрана

## Преимущества htop

1. **Цвета имеют значение:**
    - Зелёный (CPU) — обычные процессы
    - Красный (CPU) — системные процессы
    - Синий (Memory) — буферы
    - Жёлтый (Memory) — кэш
2. **Дерево процессов (F5) -** видно, кто кого запустил
3. **Фильтр (F4) -** найти конкретный процесс
4. **Убить процесс (F9) -** выбор сигнала из меню

## Полезные клавиши htop

- `F1` — справка
- `F2` — настройки (что показывать)
- `F3` — поиск процесса
- `F4` — фильтр (показать только...)
- `F5` — дерево процессов
- `F6` — выбор сортировки
- `F9` — убить процесс
- `F10` — выход

# Часть 4: Мониторинг памяти — не дайте ей закончиться

Нехватка памяти — частая причина проблем. Научимся её контролировать.

## Команда free — сколько памяти свободно

```sql
free -h
```

**Пример вывода:**

```vbnet
               total        used        free      shared  buff/cache   available
Mem:            15Gi       8.0Gi       2.1Gi       145Mi       5.9Gi       7.1Gi
Swap:          8.0Gi          0B       8.0Gi
```

## Как правильно читать вывод free

**Строка Mem (оперативная память):**

- `total` = 15Gi — всего установлено памяти
- `used` = 8.0Gi — используется приложениями
- `free` = 2.1Gi — совсем не используется (не важно!)
- `shared` = 145Mi — разделяемая память (tmpfs и др.)
- `buff/cache` = 5.9Gi — буферы и кэш
- `available` = 7.1Gi — РЕАЛЬНО доступно для программ

**Главное правило:** Смотрите на **available**, а не на free!

## **Почему free маленький, а паниковать не надо?**

Linux использует свободную память для кэширования файлов. Это ускоряет работу. Когда программе нужна память, **кэш мгновенно освобождается.**

### **Строка Swap (файл подкачки):**

- `total` = 8Gi — размер swap-раздела
- `used` = 0B — используется (должно быть 0!)
- `free` = 8Gi — свободно

### **Когда бить тревогу:**

- available меньше 10% от total
- swap used больше 0 при нормальной работе
- swap активно растёт

## Мониторинг памяти в реальном времени

```sql
watch -n 1 free -h
```

**Команда watch** повторяет любую команду с интервалом:

- `-n 1` — интервал 1 секунда
- `-d` — подсветить изменения

**Пример использования:** запустите эту команду и в другом терминале откройте браузер с 50 вкладками. Увидите, как растёт used и уменьшается available.

## Детальная информация о памяти

```bash
cat /proc/meminfo
```

**Пример вывода (сокращённо):**

```makefile
MemTotal:       16777216 kB
MemFree:         2195584 kB
MemAvailable:    7422976 kB
Buffers:          169984 kB
Cached:          6144512 kB
SwapTotal:       8388608 kB
SwapFree:        8388608 kB
```

Здесь десятки параметров. Самые важные:

- `MemTotal` — всего памяти
- `MemAvailable` — доступно для программ
- `SwapTotal/SwapFree` — swap статистика
- `Dirty` — данные, ожидающие записи на диск

**Более читаемый вариант:**

```nginx

```

**Пример вывода:**

# Часть 5: Мониторинг дисков — место и скорость

## Сколько места осталось — df

```bash
df -h
```

**Пример вывода:**

```bash
Filesystem      Size  Used Avail Use% Mounted on
/dev/sda1        50G   35G   13G  74% /
/dev/sda2       100G   45G   50G  48% /home
tmpfs           2.0G   12M  2.0G   1% /dev/shm
/dev/sdb1       500G  400G   75G  85% /data
```

**Расшифровка колонок:**

- `Filesystem` — устройство или раздел
- `Size` — общий размер
- `Used` — использовано
- `Avail` — доступно (не всегда Size - Used!)
- `Use%` — процент использования
- `Mounted on` — куда примонтировано

**На что обращать внимание:**

- `Use%` > 80% — пора чистить
- `Use%` > 90% — срочно чистить!
- `/` на 100% — система может упасть
- `/var` на 100% — логи не пишутся

**Полезные опции df:**

```bash
df -h              # человекочитаемый формат
df -i              # использование inode (для мелких файлов)
df -T              # показать тип файловой системы
df -h /home        # конкретная директория
```

## Кто съел место — du

Нашли проблемный диск? Ищем виновника:

```bash
du -h /var | sort -hr | head -20
```

**Расшифровка команды:**

- `du -h /var` — размер каждой папки в /var
- `sort -hr` — сортировка по размеру (убывание)
- `head -20` — топ-20 результатов

**Пример вывода:**

```iecst
4.5G    /var
3.8G    /var/log
2.1G    /var/log/nginx
1.5G    /var/log/nginx/access.log
512M    /var/lib
256M    /var/lib/mysql
```

**Быстрый поиск больших файлов:**

```bash
find / -type f -size +100M -exec ls -lh {} \; 2>/dev/null
```

Найдёт все файлы больше 100MB.

## **Интерактивный анализ — ncdu:**

```bash
sudo apt install ncdu
sudo ncdu /
```

Навигация как в файловом менеджере, видны размеры папок.

## Скорость дисков — iostat

Места хватает, но диски тормозят? Проверяем скорость:

```css
iostat -x 1
```

**Флаги:**

- `-x` — расширенная статистика
- `1` — обновление каждую секунду

**Пример вывода:**

```perl
avg-cpu:  %user   %nice %system %iowait  %steal   %idle
           5.03    0.00    2.01   15.58    0.00   77.39
Device     r/s     w/s     rkB/s   wkB/s   await  r_await  w_await  %util
sda       10.00   25.00   160.00  2048.00   12.50     2.00    16.67   45.20
sdb        2.00  150.00    32.00  9600.00   85.30     5.00    86.35   98.40
```

**Ключевые метрики:**

- `%iowait` — CPU ждёт диск (>20% = проблема)
- `r/s, w/s` — операций чтения/записи в секунду
- `await` — среднее время ожидания (мс)
- `%util` — загрузка диска

**Интерпретация:**

- `%util` > 70% — диск нагружен
- `%util` > 90% — диск перегружен
- `await` > 20ms — медленный отклик

В примере `sdb` загружен на **98.4%** с временем отклика `85ms` — это узкое место!

## Кто грузит диски — iotop

```bash
sudo apt install iotop
sudo iotop
```

Показывает процессы, активно использующие диски. Как top, но для **I/O.**

# Часть 6: Мониторинг сети — что и куда течёт

## Открытые порты — ss

Какие сервисы слушают сеть:

```nginx
ss -tuln
```

**Флаги:**

- `-t` — TCP порты
- `-u` — UDP порты
- `-l` — только слушающие (listening)
- `-n` — числовой формат (без DNS)

**Пример вывода:**

```ruby
State    Recv-Q   Send-Q     Local Address:Port      Peer Address:Port
LISTEN   0        128              0.0.0.0:80             0.0.0.0:*
LISTEN   0        128              0.0.0.0:22             0.0.0.0:*
LISTEN   0        128            127.0.0.1:3306           0.0.0.0:*
LISTEN   0        128                 [::]:80                [::]:*
LISTEN   0        128                 [::]:443               [::]:*
```

**Что мы видим:**

- Порт 80 — веб-сервер (HTTP)
- Порт 22 — SSH
- Порт 3306 — MySQL (только localhost!)
- Порт 443 — HTTPS

**Активные соединения:**

```nginx
ss -tn
```

Покажет, кто к кому подключён прямо сейчас.

## Трафик в реальном времени — iftop

```bash
sudo apt install iftop

```

**Что показывает iftop:**

- Соединения в реальном времени
- Скорость передачи/приёма
- Общий трафик

**Управление:**

- `n` — переключить разрешение DNS
- `s/d` — показать источник/назначение
- `p` — показать порты
- `P` — пауза
- `q` — выход

# Часть 7: Универсальный инструмент — vmstat

`vmstat` показывает всё сразу: CPU, память, диски, процессы. Это швейцарский нож мониторинга.

## Запуск и вывод

```nginx
vmstat 2
```

Обновление каждые 2 секунды.

**Пример вывода:**

### Полная расшифровка vmstat

**procs (процессы):**

- `r` — в очереди на выполнение (если > CPU = перегрузка)
- `b` — заблокированы (ждут I/O)

**memory (память в КБ):**

- `swpd` — использованный swap
- `free` — свободная память
- `buff` — буферы
- `cache` — кэш

**swap (подкачка):**

- `si` — swap in (из swap в память) КБ/сек
- `so` — swap out (из памяти в swap) КБ/сек

**io (ввод-вывод):**

- `bi` — blocks in (чтение с диска) блоков/сек
- `bo` — blocks out (запись на диск) блоков/сек

**system (система):**

- `in` — прерывания в секунду
- `cs` — переключения контекста в секунду

**cpu (процентное использование):**

- `us` — user (пользовательские процессы)
- `sy` — system (системные процессы)
- `id` — idle (простой)
- `wa` — wait (ожидание I/O)
- `st` — stolen (украдено гипервизором)

## Как интерпретировать vmstat

**Признаки проблем:**

- `r` постоянно > количества CPU — **нужно больше CPU**
- `b` часто > 0 — **проблемы с дисками**
- `si/so` не равны 0 — **не хватает памяти**
- `wa` > 20% — **диски тормозят систему**
- `cs` > 100000 — **слишком много переключений**

# Часть 8: **Пошаговая диагностика. Шпаргалка.**

1. **Запускаем top**
    
    ```css
    top
    ```
    
    Смотрим `load average`. Если больше количества ядер или близко к ним — надо смотреть.
2. **Проверяем память**
    
    ```sql
    free -h
    ```
    
    Если available < 10% или swap used > 0 — **проблемы с памятью.**
3. **Проверяем диски**
    
    ```css
    iostat -x 1
    ```
    
    Если %util > 90% или await > 50ms — **диски не справляются.**
4. **Смотрим сеть**
    
    ```undefined
    iftop
    ```
