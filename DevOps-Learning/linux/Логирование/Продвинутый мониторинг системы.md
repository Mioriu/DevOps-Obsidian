Если базовый мониторинг — это измерение температуры и пульса, то продвинутый — это полное медицинское обследование с МРТ, анализами крови и кардиограммой. Сегодня мы научимся видеть систему насквозь.

### Когда нужны продвинутые инструменты?

- **Кейс 1:** Веб-сайт периодически **подвисает** на 2-3 секунды. **`top`** показывает нормальную загрузку. Нужно копать глубже.
- **Кейс 2:** Приложение работает **медленно**, но **CPU и память в норме.** Где искать проблему?
- **Кейс 3:** После обновления производительность **упала на 30%**. Как найти причину?

## Для начала Strace — рентген для процессов

`**strace**` показывает **ВСЕ** системные вызовы процесса. Это как подслушивать разговор программы с операционной системой.

### Простой пример — что делает команда ls?

```bash
strace ls
```

Вы увидите сотни строк! Каждая — это системный вызов:

```erlang
openat(AT_FDCWD, ".", O_RDONLY|O_NONBLOCK|O_CLOEXEC|O_DIRECTORY) = 3
getdents64(3, /* 23 entries */, 32768) = 696
close(3) = 0
write(1, "Desktop  Documents  Downloads\n", 30) = 30
```

### Базовое использование

- Трассировка команды — `strace ls -la`
- Трассировка запущенного процесса — `sudo strace -p $(pidof nginx)`

### Полезные опции strace

Подсчёт системных вызовов:

```bash
strace -c ls -la
```

Фильтрация по типу вызовов:

```bash
# Только файловые операции
strace -e file ls -la

# Только сетевые операции
strace -e network curl https://example.com

# Только операции с памятью
strace -e memory firefox
```

Сохранение вывода в файл с временными метками:

```lua
strace -tt -o output.log nginx
```

## Практическое применение strace

### **Сценарий 1: Программа не запускается, ошибок нет**

```perl
strace -e open,openat myprogram 2>&1 | grep -E "ENOENT|EACCES"
```

Покажет, какие файлы программа не может найти (`ENOENT`) или открыть (`EACCES` — нет прав).

### **Сценарий 2: Программа зависает**

```bash
# Подключаемся к работающему процессу
sudo strace -p $(pidof myprogram)
```

Если видите что-то вроде:

```sql
select(5, [4], NULL, NULL, NULL
```

Программа ждёт данные из сети (дескриптор 4). **Проблема в сети**!

### **Сценарий 3: Подсчёт системных вызовов**

```bash
strace -c ls
```

Покажет статистику:

```lua
% time     seconds  usecs/call     calls    errors syscall
------ ----------- ----------- --------- --------- ----------------
 23.45    0.000234           5        45           write
 18.32    0.000183           4        41           close
 15.67    0.000156           7        22         2 openat
```

## Фильтры strace — находим нужное

**`strace`** генерирует МНОГО информации. Используем **фильтры**:

- `-e file` — только файловые операции
- `-e network` — только сетевые операции
- `-e process` — создание процессов
- `-e memory` — работа с памятью

**Пример: отладка веб-сервера**

```graphql
# Смотрим только сетевую активность
sudo strace -e network -p $(pidof nginx)
# Видим:
accept4(6, {sa_family=AF_INET, sin_port=htons(54321), sin_addr=inet_addr("192.168.1.100")}, [112->16], SOCK_CLOEXEC) = 7
recvfrom(7, "GET / HTTP/1.1\r\nHost: example.c"..., 8192, 0, NULL, NULL) = 326
sendto(7, "HTTP/1.1 200 OK\r\nServer: nginx/1"..., 238, 0, NULL, 0) = 238
```

Видно: **принял соединение, получил запрос, отправил ответ.**

# perf — профилирование производительности

**perf** — это мощный инструмент для анализа производительности на уровне ядра Linux.

## Установка perf

```bash
sudo apt install linux-tools-common linux-tools-generic
```

## Запись профиля производительности

### **Шаг 1: Записываем активность**

```bash
# Записываем 30 секунд работы системы
sudo perf record -a -g sleep 30
Или профилируем конкретную команду
sudo perf record -g ./myprogram
```

### **Шаг 2: Анализируем результаты**

```bash
sudo perf report
```

Увидите интерактивный отчёт. Можно проваливаться в функции и смотреть, кто их вызывает.

```bash
sudo perf record -a -g sleep 10
```

## Мониторинг событий в реальном времени

```bash
sudo perf top
```

Вы увидите "горячие" функции в реальном времени:

```css
Overhead  Shared Object       Symbol
  12.34%  [kernel]           [k] copy_user_generic_string
   8.76%  libc-2.31.so       [.] __memcpy_avx_unaligned
   5.43%  myapp              [.] process_data
```

В этом примере 12% времени CPU тратится на копирование данных в ядре.

## Анализ конкретного процесса

```bash
# Найдём PID процесса
pidof nginx

# Профилируем его
sudo perf record -p $(pidof nginx) -g sleep 30
sudo perf report
```

## Практический пример: оптимизация веб-приложения

```bash
# Профилируем веб-сервер под нагрузкой
sudo perf record -p $(pidof python3) -g sleep 60

# Смотрим результаты
sudo perf report

# Видим:
# 35% времени - функция serialize_json
# 20% времени - запросы к БД
# 15% времени - сжатие ответов
```

Теперь знаем, что оптимизировать!

# Мониторинг давления на систему (PSI)

С версии ядра 4.20 в Linux появился новый механизм — PSI (Pressure Stall Information). Это показатель давления на подсистемы.

## Что такое "давление"?

Представьте пробку на дороге:

- Если 10% времени машины стоят — небольшая пробка
- Если 50% времени — серьёзная пробка
- Если 90% времени — дорога встала

`**PSI**` показывает такие **пробки** для `CPU, памяти и дисков.`

## Как читать PSI

```bash
cat /proc/pressure/memory
```

Вывод:

```sql
some avg10=0.00 avg60=5.25 avg300=3.14 total=1234567
full avg10=0.00 avg60=2.10 avg300=1.05 total=567890
```

Расшифровка:

- `some` — хотя бы один процесс ждал память
- `full` — ВСЕ процессы ждали память
- `avg10` — процент времени ожидания за последние 10 секунд
- `avg60` — за последнюю минуту
- `avg300` — за последние 5 минут

# glances — всё в одном месте

**glances** — это как кабина пилота: все приборы на одном экране.

## Установка и запуск

```bash
sudo apt install glances
glances
```

## Режимы работы glances

### **1. Локальный мониторинг**

```nginx
glances
```

### **2. Веб-интерфейс**

```shell
glances -w
# Открыть в браузере: http://server-ip:61208
```

```cmake
# В CSV для Excel
```

# Дополнительные инструменты

## Исторический анализ с sar

**sar** (System Activity Reporter) собирает и хранит статистику производительности.

### Установка и настройка

```bash
sudo apt install sysstat
```

Включите сбор данных:

```bash
sudo vim /etc/default/sysstat
# Измените ENABLED="false" на ENABLED="true"

sudo systemctl enable sysstat
sudo systemctl start sysstat
```

### Использование sar

- CPU статистика за сегодня: `sar -u`
- Память: `sar -r`
- Сеть: `sar -n DE`
- Диски: `sar -d`

Исторические данные:

```bash
# За вчера
sar -u -f /var/log/sysstat/sa$(date -d yesterday +%d)

# За определённый период
sar -u -s 10:00:00 -e 14:00:00
```
